## 1. Что такое успешный AI-ассистент?

| Участник | Позиция |
|----------|---------|
| **Наиль** | Ориентируется в личных данных, отвечает на вопросы типа "когда я впервые обсуждал проект X", безопасный "Джарвис" |
| **Юрий** | Единый архив всех коммуникаций с распознаванием спикеров, голосовой интерфейс, понимание контекста жизни человека |
| **Евгений** | Персональная база знаний с LLM-запросами (умный поиск), on-premise архитектура |
| **Миша** | Суверенное локальное решение, глубоко владеющее контекстом, ведёт диалог в Telegram |
| **Алексей** | Структурированное понимание человека (7 слоёв), отслеживание динамики связей |
| **Данил** | Присвоение мнений авторам, выявление расхождений во взглядах, определение ролей |

### Точки схождения

- Локальность и суверенность
- Работа с личными данными (переписки, встречи)
- Понимание контекста

### Точки расхождения

- **Наиль**: фокус на быстрой ценности
- **Юрий**: фокус на полноте архива
- **Алексей**: фокус на структуре "Человека"

## 2. Что делать первым?

| Участник | Позиция |
|----------|---------|
| **Наиль** | Развернуть бэкап телеги + простую LLM, потестить, понять что не хватает. Минимальные итерации с ценностью |
| **Юрий** | Распознавание спикеров + справочник контактов. Сначала разметить голоса и людей |
| **Евгений** | Транскрибация без распознания персон + саммари (3-4 недели), потом распознание персон (+2-3 месяца) |
| **Миша** | whisper + pyannote + LLM + векторная база. Собрать pipeline обработки |
| **Алексей** | Определить структуру человека → наполнить одного человека → научить отвечать → только потом расширять |
| **Данил** | Взять нашу компанию как "владельца", обучить на чатах, проверить на реальных вопросах |

### Точки схождения

- Начать с малого, расширять постепенно
- Telegram-бэкапы как источник данных

### Точки расхождения

- **Наиль/Миша**: быстрый старт с готовыми решениями
- **Юрий**: сначала распознать спикеров
- **Алексей**: сначала структура человека
- **Евгений**: транскрибация первична

## 3. Как понять, что получилось?

| Участник | Позиция |
|----------|---------|
| **Наиль** | Связать технические метрики с продуктовыми. Реальное использование = критерий. "Довести до реального UX" |
| **Юрий** | Нейронка узнаёт людей по голосу, диалоги разложены на авторов, можно оперировать именами собеседников |
| **Евгений** | Практическая ценность: "мы уже регулярно пользуемся ботом для обработки записей встреч" |
| **Миша** | Ассистент глубоко владеет контекстом, отвечает на вопросы, ведёт диалог |
| **Алексей** | Корректные ответы на базовые вопросы по одному человеку. "Без этого весь ассистент будет бесполезен" |
| **Данил** | Модель присваивает мысли участникам, видит различия, определяет расхождение взглядов |

### Точки схождения

- Практическое использование важнее метрик
- Корректность ответов

### Точки расхождения

- **Юрий**: распознавание голоса как критерий
- **Данил**: атрибуция мнений как критерий
- **Алексей**: базовые вопросы по одному человеку

## 4. Какие ограничения учитывать?

| Участник | Позиция |
|----------|---------|
| **Наиль** | Избегать сложного RND. Опасение: "ценный результат пощупаем только через много месяцев" |
| **Юрий** | Качество зависит от знания спикеров заранее. Нужен справочник контактов |
| **Евгений** | Нельзя перенести "магию" облака на коробку. Голосовые слепки ненадёжны. Нужны правильные запросы от пользователя |
| **Миша** | RAG может вредить: "подтягивает нерелевантную информацию, создавая галлюцинации" |
| **Алексей** | Нельзя расширять контекст, пока не работает базовое. "Не доделали шаг 3 и перешли на шаг 4" |
| **Данил** | — |

### Точки схождения

- Не переоценивать возможности локальных LLM
- Сначала базовое, потом расширение

### Точки расхождения

- **Наиль**: ограничение по времени/затратам
- **Евгений**: ограничение по мощности железа
- **Миша**: ограничение RAG-архитектуры

## 5. Как AI должен понимать контекст?

| Участник | Позиция |
|----------|---------|
| **Наиль** | Разные участники видят проект по-разному. AI должен выявлять расхождения в понимании |
| **Юрий** | AI должен понимать контекст жизни человека. "Человек в контексте" — ключевое отличие от AI |
| **Евгений** | Модель знает только то, что загружено. Ценность растёт со временем |
| **Миша** | Субъективные ответы от лица владельца. RAG + граф знаний |
| **Алексей** | 7 слоёв человека + субъективные связи. Динамика изменений: кто стал важнее, что перестало быть актуальным |
| **Данил** | Присвоение мнений авторам, иерархия приоритетов высказываний |

### Точки схождения

- Субъективность: разные взгляды разных людей
- Контекст = понимание человека

### Точки расхождения

- **Алексей**: структурированные слои
- **Юрий**: "контекст жизни"
- **Данил**: иерархия приоритетов

## Ключевые расхождения

### 1. Что первично: технология или структура?

- **Технология first**: Наиль, Миша, Евгений — начать с готовых решений
- **Структура first**: Алексей, Юрий — сначала понять "человека"

### 2. Как измерять успех?

- **Практическое использование**: Наиль, Евгений
- **Технические метрики**: Юрий (распознание голоса)
- **Концептуальные**: Данил (атрибуция), Алексей (базовые вопросы)

### 3. Что такое "контекст"?

- **Наиль/Данил**: разные взгляды участников проекта
- **Юрий**: контекст всей жизни человека
- **Алексей**: структурированные слои + связи
- **Евгений**: только загруженные данные

---

## Зоны консенсуса

1. **Суверенность** — данные не уходят в облако (все согласны)
2. **Итеративность** — двигаться шагами, проверять гипотезы (все согласны)
3. **Telegram-бэкапы** — основной источник данных на старте (все согласны)
4. **Субъективность** — AI должен понимать разные точки зрения (Наиль, Алексей, Данил)
5. **Практическая ценность** — важнее чем технические метрики (Наиль, Евгений)
